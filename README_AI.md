# 🤖 AI 작업 및 협업 규칙 (사용자: 코드 미숙련자)

## 1. 기본 원칙
- **안정성 최우선:** 사용자는 코드를 직접 수정하거나 읽는 데 익숙하지 않다. 따라서 AI는 기존에 작동하던 기능을 절대 깨뜨리지 않아야 한다.
- **보고 필수:** 수정을 시작하기 전, 어떤 파일을 고칠 것인지와 예상되는 변화를 한국어로 요약 보고하라.
- **설명 방식:** 기술 용어보다는 "어느 화면의 어떤 버튼이 어떻게 바뀐다"는 식으로 사용자 관점에서 설명하라.

## 2. UI/UX 디자인 가이드 (시인성 및 고급화)
- **참조:** 모든 디자인 수정은 기존의 시각적 테마와 'Philosophia' 브랜드 아이덴티티를 최우선으로 참고한다.
- **일관성:** 특정 스타일(색상, 여백, 둥글기 등)을 변경할 때 프로젝트 전체의 통일성을 유지하라.
- **시인성:** 텍스트 가독성을 높이기 위해 적절한 대비와 여백(Padding/Margin)을 충분히 사용하라.
- **CSS 규칙:** 이 프로젝트는 **Vanilla CSS**(`style.css`)를 사용한다. 기존에 정의된 CSS 변수(`--gold`, `--bg` 등)와 클래스 구조를 준수하며, 파괴적인 변경을 하지 마라.

## 3. 기능 및 로직 보호 (Firebase 관련)
- **수정 제한:** `firebase.json` 설정 파일이나 인증(Auth), 데이터베이스 관련 로직은 기능 추가가 아닌 이상 최대한 건드리지 마라.
- **파괴적 수정 금지:** 변수 이름을 바꾸거나 파일 구조를 갈아엎는 행위는 절대 금지한다.
- **승인 절차:** 만약 로직 수정이 불가피하다면, "어떤 위험이 있는지" 먼저 설명하고 사용자의 승인을 받은 뒤 진행하라.

## 4. 작업 완료 후 체크리스트
- 작업이 끝나면 사용자가 직접 테스트해 볼 수 있도록 다음 정보를 제공하라.
  1. 수정한 파일 목록
  2. 눈으로 확인할 수 있는 디자인 변경 사항
  3. 정상 작동 여부를 확인해야 할 주요 버튼이나 링크 목록

## 5. 문제 발생 시 대처
- 만약 사용자가 "기능이 안 된다"거나 "화면이 이상하다"고 하면, 즉시 이전 커밋(Commit) 상태를 참고하여 복구 방안을 제시하라.
